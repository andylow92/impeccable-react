# impeccable-react

A production-grade React + TypeScript starter for teams that care about **craft**, not autogenerated SaaS sameness.

## Why most AI UIs look generic
Most AI-generated interfaces converge on the same weak defaults:
- Rounded cards + soft shadows + cosmetic gradients.
- Gray-on-gray text that hides priority.
- Components that look individually acceptable but lack compositional hierarchy.
- Progress bars that imply motion while communicating almost nothing.

This repository intentionally rejects those defaults.

## Before vs After (Design direction)
### Before (generic AI output)
- Flat dashboard cards with identical radii and shadows.
- Weak typography where headings and values feel interchangeable.
- Decorative progress bars with little accountability.

### After (this starter)
- Card structure with an explicit heading system and decision zones.
- High-contrast typography and role-based color choices.
- Progress represented as a status sequence with owner visibility.
- Spacing based on repeatable rhythm tokens, not arbitrary padding.

## Stack
- React + TypeScript (strict)
- Vite
- Tailwind CSS (lightweight utility layer)
- Zod boundary validation for external data

## Project structure
```txt
src/
  domain/   # validated domain contracts and types
  lib/      # typed API + shared utilities
  ui/       # components + screens
skills/
  design.md # reusable anti-generic UI skill
```

## TypeScript discipline
- `strict` mode and `noImplicitAny` enabled.
- `@typescript-eslint/no-explicit-any` enforced.
- External data parsed with Zod before entering domain/UI.
- API functions return typed domain objects.

## Component system (non-generic by design)
- `Button`: sharp, high-contrast action styles (no anonymous pill defaults).
- `Card`: structured heading + content zones, not decorative container chrome.
- `ProgressSignal`: owner-aware status rail instead of a generic bar.

## Impeccable design principles adapted to React
1. **Hierarchy before decoration**: ensure a user can parse importance in seconds.
2. **Contrast is functional**: color encodes state and action, not visual noise.
3. **Rhythm scales quality**: spacing is a system, not hand-tuned magic numbers.
4. **Components explain themselves**: each UI primitive documents what generic pattern it avoids.

## Quick start

